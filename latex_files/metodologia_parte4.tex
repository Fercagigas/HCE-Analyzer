\subsection{Capa de inteligencia artificial}
\label{sec:stack_ia}

El núcleo de IA se articula en torno a la API de Anthropic con tres modelos
en cadena de \textit{fallback}:

\begin{enumerate}[nosep, leftmargin=*]
  \item \textbf{Primario}: \texttt{claude-haiku-4-5-20251001} — modelo de baja
    latencia y coste reducido, configurado con \texttt{max\_tokens=4096} y
    \texttt{temperature=0.1}. Gestiona el 95\,\% de las consultas en producción.
  \item \textbf{Secundario}: \texttt{claude-sonnet-4-5-20250929} — activado
    automáticamente ante errores de sobrecarga del modelo primario.
    Empleado también como modelo exclusivo del \texttt{VisualizationAgent}
    (\texttt{max\_tokens=8192}) por su mayor capacidad de generación de código.
  \item \textbf{Terciario}: \texttt{claude-opus-4-20250514} — reserva de máxima
    capacidad para consultas de alta complejidad o fallos en cascada.
\end{enumerate}

La integración con LangChain se realiza mediante \texttt{langchain-anthropic}
$\geq$\,0.3.0, que expone la clase \texttt{ChatAnthropic} con soporte nativo
de \textit{tool calling} estructurado. El conteo de tokens se gestiona con
\texttt{tiktoken} $\geq$\,0.7.0 para optimizar el uso del contexto disponible.
Los \textit{embeddings} para el sistema RAG se generan con el modelo
\texttt{sentence-transformers/all-MiniLM-L6-v2} (384 dimensiones), cargado
localmente mediante \texttt{sentence-transformers} $\geq$\,3.0.0.

\subsection{Capa de datos y persistencia}
\label{sec:stack_datos}

Supabase actúa como plataforma de datos unificada con dos esquemas diferenciados:

\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{Esquema \texttt{public}}: tablas de aplicación (\texttt{users},
    \texttt{chat\_sessions}, \texttt{chat\_messages}, \texttt{clinical\_documents})
    con Row Level Security habilitado en las 13 tablas del sistema.
  \item \textbf{Esquema \texttt{mimic\_ed}}: seis tablas del dataset MIMIC-IV-ED
    (\texttt{edstays}, \texttt{triage}, \texttt{vitalsign}, \texttt{diagnosis},
    \texttt{medrecon}, \texttt{pyxis}) con 31 índices B-tree optimizados para
    consultas por \texttt{subject\_id}, \texttt{stay\_id} y \texttt{charttime}.
\end{itemize}

La extensión \texttt{pgvector} de PostgreSQL almacena los \textit{embeddings}
de los documentos clínicos indexados, permitiendo búsqueda vectorial por
similitud coseno directamente en la base de datos sin infraestructura adicional.
El caché de respuestas LLM y de \textit{embeddings} se implementa con Redis,
con TTL configurable de 300\,s por defecto (\texttt{CACHE\_TTL\_SECONDS}).

\subsection{Capa de presentación y procesamiento}
\label{sec:stack_frontend}

La interfaz de usuario se construye con Streamlit, que permite desplegar
aplicaciones web interactivas en Python sin separación frontend/backend.
El punto de entrada es \texttt{main.py}, que carga \texttt{src/core/app.py}
y renderiza \texttt{ui/unified\_chat\_interface.py} como componente principal.
FastAPI con Uvicorn proporciona una API REST opcional para integración con
sistemas externos. El procesamiento de documentos clínicos utiliza Docling
para extracción avanzada de PDF y \texttt{python-docx} para documentos Word.
Las visualizaciones se generan con Plotly (gráficas interactivas HTML) y
Matplotlib (imágenes estáticas PNG a 150\,DPI).

\subsection{Entorno de desarrollo y calidad de código}
\label{sec:entorno_desarrollo}

El proyecto requiere Python 3.8+ y gestiona dependencias mediante
\texttt{requirements.txt} con versiones mínimas especificadas.
El entorno recomendado es Conda (\texttt{environment.yml}).
Las herramientas de calidad de código incluyen Black para formateo automático,
Flake8 para análisis estático con longitud máxima de línea de 100 caracteres,
y pytest con pytest-asyncio para la suite de pruebas.
El logging estructurado se implementa con \texttt{structlog} y el módulo
estándar \texttt{logging}, con rotación automática de archivos de 10\,MB
y retención de 30 días.

\section{Marco de evaluación}
\label{sec:evaluacion}
% ------------------------------------------------------------------

La evaluación de ChatHCE se estructura en cuatro dimensiones complementarias
que cubren calidad de respuesta, rendimiento temporal, robustez ante ataques
y corrección funcional. El marco se implementa en el directorio
\texttt{Evaluation/} con cinco módulos independientes orquestados por
\texttt{run\_all\_evaluations.py}. Todos los módulos instancian el agente
mediante la función \texttt{instantiate\_agent()} del módulo compartido
\texttt{eval\_helpers.py}, que crea una instancia de \texttt{UnifiedChatAgent}
en condiciones idénticas a producción.

\subsection{Evaluación RAGAS: calidad de recuperación y generación}
\label{sec:eval_ragas}

RAGAS (\textit{Retrieval-Augmented Generation Assessment Score}) es un
\textit{framework} de evaluación sin referencia humana que mide la calidad
de sistemas RAG mediante cuatro métricas complementarias
\cite{es2023ragas}:

\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{Faithfulness} ($\geq 0.85$): mide si cada afirmación de la
    respuesta puede inferirse del contexto recuperado, sin introducir
    información externa. Se calcula como la proporción de afirmaciones
    verificables sobre el total de afirmaciones de la respuesta.
  \item \textbf{Answer Relevancy} ($\geq 0.80$): evalúa si la respuesta
    aborda directamente la pregunta formulada, penalizando respuestas
    incompletas o que desvían el foco. Se estima generando preguntas
    sintéticas a partir de la respuesta y midiendo su similitud semántica
    con la pregunta original.
  \item \textbf{Context Precision} ($\geq 0.75$): mide qué proporción de
    los fragmentos recuperados son realmente relevantes para responder la
    pregunta, penalizando el ruido en el contexto.
  \item \textbf{Context Recall} ($\geq 0.70$): evalúa si el contexto
    recuperado contiene toda la información necesaria para generar la
    respuesta de referencia (\textit{ground truth}).
\end{itemize}

\subsubsection{Golden set de evaluación}
\label{sec:golden_set}

El golden set de base de datos (\texttt{golden\_set\_ragas.json}) contiene
40 preguntas con respuestas de referencia verificadas directamente contra
la base de datos MIMIC-IV-ED. La distribución por categoría es:
8 preguntas de resumen de paciente (\texttt{patient\_summary}),
8 de signos vitales (\texttt{vital\_signs}),
8 de diagnósticos (\texttt{diagnoses}),
8 de medicamentos (\texttt{medications}),
4 de triaje (\texttt{triage}) y
4 de consultas multi-tabla (\texttt{cross\_table}).
Cada pregunta incluye la consulta SQL de referencia
(\texttt{ground\_truth\_sql}) y los fragmentos de contexto esperados,
lo que permite verificación determinista de las respuestas.

El golden set RAG (\texttt{golden\_set\_ragas\_rag.json}) contiene
30 preguntas distribuidas en cinco tipos: directa (9), multi-fragmento (6),
sinónimo (6), razonamiento aplicado (6) y fuera de dominio (3).
Las preguntas se generan a partir de tres documentos clínicos reales:
el \textit{Manual del Residente de Medicina Intensiva} del Hospital
Universitario Virgen del Rocío, los \textit{Estándares y Recomendaciones
para Unidades de Cuidados Intensivos} del Ministerio de Sanidad, y
\textit{El Libro de la UCI} de Paul L. Marino (3.ª edición).
Las respuestas de referencia se generan mediante el método
\textit{LLM-as-Extractor} con temperatura cero y un \textit{prompt}
restrictivo que limita la respuesta exclusivamente al contenido de los
fragmentos identificados como relevantes.

\subsubsection{Implementación técnica}
\label{sec:ragas_impl}

El script \texttt{run\_ragas\_eval.py} implementa el pipeline de evaluación
en cuatro etapas. Primero, carga y valida el golden set verificando la
presencia de los campos obligatorios (\texttt{id}, \texttt{question},
\texttt{ground\_truth}, \texttt{ground\_truth\_sql}, \texttt{contexts},
\texttt{category}). Segundo, invoca \texttt{UnifiedChatAgent.process\_message()}
para cada pregunta con reintentos automáticos ante errores de límite de tasa
(\texttt{retry\_on\_rate\_limit}, máximo 3 intentos con espera de 60\,s).
Tercero, extrae los contextos de la respuesta combinando las fuentes RAG
(\texttt{sources}) y los resultados de herramientas (\texttt{tool\_results}).
Cuarto, construye un \texttt{datasets.Dataset} compatible con RAGAS 0.4.3
y ejecuta \texttt{evaluate()} con las cuatro métricas.

\subsubsection{Resultados}
\label{sec:ragas_resultados}

La ejecución del 12 de marzo de 2026 procesó las 40 preguntas del golden set
de base de datos en 1177\,s sin errores (0 fallos de invocación al agente).
La librería RAGAS 0.4.3 no completó el cálculo de métricas agregadas en esta
ejecución por un error interno de la librería al procesar los contextos
generados por el agente, lo que impidió obtener puntuaciones numéricas
finales. No obstante, el 100\,\% de las preguntas recibieron respuesta del
agente (\texttt{Status: OK} en los 40 casos), confirmando la disponibilidad
y estabilidad del sistema durante la evaluación.

\begin{tcolorbox}[
  colback=secondary!5,
  colframe=secondary!60,
  title={\textbf{Nota metodológica — Evaluación RAGAS}},
  fonttitle=\small\bfseries,
  left=6pt, right=6pt, top=4pt, bottom=4pt
]
\small
La imposibilidad de obtener puntuaciones RAGAS agregadas se debe a una
incompatibilidad entre la versión 0.4.3 de la librería y el formato de

de herramientas de base de datos). Esta limitación es conocida en la
comunidad RAGAS para sistemas multi-herramienta y no refleja un fallo del
sistema evaluado. La evaluación funcional mediante casos de prueba
(Sección~\ref{sec:eval_funcional}) proporciona evidencia complementaria
de la calidad de las respuestas.

\subsection{Casos de prueba funcionales}
\label{sec:eval_funcional}

El script \texttt{run\_test\_cases.py} implementa 28 casos de prueba
funcionales distribuidos en cuatro categorías: 10 de base de datos (TC-DB),
8 de recuperación de documentos (TC-RAG), 5 de visualización (TC-VIZ) y
5 de comportamiento del agente (TC-AGENT). Cada caso de prueba se evalúa
mediante criterios ponderados que producen una puntuación agregada
$s_w \in [0, 1]$:

\begin{equation}
s_w = \frac{\displaystyle\sum_{i} w_i \cdot s_i}{\displaystyle\sum_{i} w_i}
\label{eq:weighted_score}
\end{equation}

donde $w_i$ es el peso del criterio $i$ y $s_i \in [0, 1]$ su puntuación.
Un caso de prueba se considera aprobado cuando $s_w \geq s_{\min}$, con
$s_{\min}$ definido individualmente por caso (rango: 0.50--0.60).
Los cuatro criterios de evaluación son:

\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{contiene\_valor} (peso 0.25--0.40): verifica que la respuesta
    contenga los valores esperados (IDs de paciente, términos clínicos).
  \item \textbf{herramienta\_correcta} (peso 0.25--0.40): comprueba que el
    agente invocó la herramienta esperada (\texttt{query\_mimic\_database},
    \texttt{search\_clinical\_documents} o \texttt{generate\_visualization}).
  \item \textbf{no\_alucinacion} (peso 0.20--0.40): detecta indicadores de
    información fabricada y verifica el manejo correcto de casos fuera de
    dominio y datos inexistentes.
  \item \textbf{formato\_respuesta} (peso 0.10--0.35): evalúa que la respuesta
    esté en español y tenga una longitud adecuada ($>$100 caracteres).
  \item \textbf{fuentes\_citadas} (peso 0.15, solo TC-RAG): verifica que la
    respuesta incluya referencias a los documentos fuente.
\end{itemize}

\texttt{generate\_visualization} en la lista \texttt{tools\_used} de la
respuesta, lo que constituye una limitación del evaluador, no del sistema.

\subsubsection{Análisis de criterios transversales}
\label{sec:criterios_transversales}

El criterio \texttt{no\_alucinacion} obtuvo puntuación 1.0 en los 28 casos
de prueba, confirmando la efectividad de las directivas anti-alucinación
implementadas en el \textit{system prompt} del agente. El criterio
\texttt{formato\_respuesta} obtuvo 1.0 en 26 de los 28 casos, con las dos
excepciones en TC-VIZ-003 (respuesta demasiado corta por fallo de herramienta)
y TC-VIZ-005 (puntuación parcial). El criterio \texttt{fuentes\_citadas}
obtuvo 1.0 en los siete casos RAG que lo incluyen, confirmando que el sistema
cita consistentemente los documentos fuente en las respuestas RAG.

\subsection{Orquestación de la evaluación}
\label{sec:eval_orquestacion}

El script \texttt{run\_all\_evaluations.py} orquesta la ejecución secuencial
de los cuatro módulos de evaluación, realizando verificaciones previas
(\textit{pre-flight checks}) que incluyen: existencia de los ficheros del
golden set, disponibilidad de la variable de entorno \texttt{ANTHROPIC\_API\_KEY},
conectividad con Supabase y existencia del directorio de resultados.
Los resultados de cada módulo se escriben en ficheros TXT con nomenclatura
\texttt{\{módulo\}\_results\_\{YYYYMMDD\_HHMMSS\}.txt} en el directorio
\texttt{Evaluation/results/}. La ejecución completa del 12 de marzo de 2026
tardó 2750.6\,s (aproximadamente 45 minutos) y los cuatro módulos completaron
sin errores de ejecución, con el resultado consolidado de PASS en el informe
\texttt{consolidated\_report\_20260312\_155759.txt}.

\section{Limitaciones y trabajo futuro}
\label{sec:limitaciones}
% ------------------------------------------------------------------

\begin{tcolorbox}[
  colback=orange!5,
  colframe=orange!70!black,
  title={\textbf{Limitaciones identificadas del sistema}},
  fonttitle=\small\bfseries,
  left=6pt, right=6pt, top=4pt, bottom=4pt
]
\small
Esta sección documenta de forma honesta las limitaciones actuales del sistema
ChatHCE, distinguiendo entre limitaciones del dataset, del sistema de evaluación
y de la arquitectura técnica.
\end{tcolorbox}

\subsection{Limitaciones del dataset MIMIC-IV-ED}
\label{sec:lim_dataset}

El dataset MIMIC-IV-ED utilizado es la versión de demostración 2.2, que
contiene 64 pacientes únicos con 222 estancias en urgencias. Esta escala
reducida, aunque suficiente para validar la arquitectura del sistema, no
es representativa de la variabilidad clínica de un servicio de urgencias
real. Las principales limitaciones derivadas del dataset son:

\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{Escala reducida}: 64 pacientes frente a los más de 400\,000
    de la versión completa de MIMIC-IV-ED. Las consultas de análisis
    estadístico (distribuciones, frecuencias) producen resultados con
    escasa significación estadística.
  \item \textbf{Datos históricos}: los registros corresponden a estancias
    entre los años 2125 y 2201 (fechas anonimizadas), lo que impide
    cualquier análisis de tendencias temporales reales.
  \item \textbf{Ausencia de datos clínicos completos}: el dataset de
    demostración no incluye notas clínicas, resultados de laboratorio
    ni imágenes diagnósticas, limitando el alcance del análisis clínico.
  \item \textbf{Datos de urgencias exclusivamente}: el sistema no tiene
    acceso a información de hospitalización, consultas externas ni
    seguimiento ambulatorio de los pacientes.
\end{itemize}

\subsection{Limitaciones del sistema de evaluación}
\label{sec:lim_evaluacion}

La evaluación RAGAS no pudo completar el cálculo de métricas agregadas
en la ejecución del 12 de marzo de 2026 por una incompatibilidad entre
RAGAS 0.4.3 y el formato de contextos multi-herramienta del agente.
Las puntuaciones de faithfulness, answer relevancy, context precision y
context recall no están disponibles como valores numéricos verificados.

Los falsos negativos en las pruebas de seguridad (SEC-SQL-005 y SEC-PROMPT-002)
revelan que las funciones de verificación basadas en listas de palabras clave
son frágiles ante variaciones en el texto de respuesta del agente. Un sistema
de verificación más robusto requeriría clasificadores semánticos o evaluación
humana.

El golden set RAG fue generado mediante el método \textit{LLM-as-Extractor}
sin validación por expertos clínicos, lo que introduce un sesgo potencial
al ser el mismo tipo de modelo que el sistema evaluado. Las preguntas de
tipo razonamiento aplicado y fuera de dominio son las más susceptibles a
este sesgo.

\subsection{Limitaciones técnicas de la arquitectura}
\label{sec:lim_tecnicas}

\begin{itemize}[nosep, leftmargin=*]
  \item \textbf{Dependencia de API externa}: el sistema requiere conectividad
    con la API de Anthropic para todas las consultas. Un fallo o degradación
    del servicio impacta directamente en la disponibilidad del sistema.
    La cadena de \textit{fallback} mitiga parcialmente este riesgo.
  \item \textbf{Registro de herramientas en TC-VIZ}: el \texttt{VisualizationAgent}
    no registra \texttt{generate\_visualization} en \texttt{tools\_used},
    lo que impide la verificación automática del uso de la herramienta de
    visualización en los casos de prueba funcionales.
  \item \textbf{Escalabilidad del sistema RAG}: la versión actual indexa
    documentos en una única colección (\texttt{clinical\_guidelines}) sin
    particionado por especialidad. Con un volumen elevado de documentos,
    la precisión del contexto recuperado puede degradarse.
  \item \textbf{Sesión de usuario única}: la arquitectura Streamlit actual
    no soporta múltiples usuarios concurrentes de forma nativa. El despliegue
    en producción requeriría un servidor Streamlit con gestión de sesiones
    o migración a una arquitectura FastAPI con frontend desacoplado.
\end{itemize}

\subsection{Líneas de trabajo futuro}
\label{sec:trabajo_futuro}

Las principales líneas de mejora identificadas a partir de los resultados
de evaluación son:

\begin{enumerate}[nosep, leftmargin=*]
  \item \textbf{Integración con MIMIC-IV completo}: migrar a la versión
    completa del dataset (acceso mediante PhysioNet) para obtener resultados
    estadísticamente significativos y mayor cobertura clínica.
  \item \textbf{Evaluación RAGAS con versión compatible}: actualizar a
    RAGAS $\geq$\,0.5.0 o implementar un adaptador de contextos para
    obtener métricas de faithfulness y relevancia verificadas.
  \item \textbf{Corrección del registro de herramienta de visualización}:
    añadir \texttt{generate\_visualization} a \texttt{tools\_used} en el
    \texttt{VisualizationAgent} para habilitar la verificación automática.
  \item \textbf{Evaluación humana}: incorporar evaluación por médicos de
    urgencias para validar la corrección clínica de las respuestas, especialmente
    en consultas de interpretación de signos vitales y diagnósticos.
  \item \textbf{Particionado del índice RAG}: implementar colecciones
    separadas por especialidad médica para mejorar la precisión del contexto
    recuperado en consultas especializadas.
  \item \textbf{Soporte multi-usuario}: migrar la interfaz a una arquitectura
    FastAPI + React o FastAPI + Streamlit con gestión de sesiones para
    soportar múltiples usuarios concurrentes en entornos hospitalarios reales.
\end{enumerate}
